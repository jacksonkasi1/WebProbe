This document defines **non-negotiable standards** for a WebSeed website: UX, mobile responsiveness, accessibility, SEO, performance, security, and quality.  
A “check” tool should be able to audit a live URL and/or source code against these standards and produce:

- A prioritized issue list (severity + evidence + where/how to fix)
- Screenshots (mobile/tablet/desktop)
- Optional “fix plan” patches (only applied after explicit confirmation)

---

## 1) Required project metadata (must be confirmed)

These must be asked/confirmed by the tool because code often contains stale or incorrect values.

**Project Identity**
- `site_name` (human name)
- `brand_name` (if different)
- `primary_domain` (canonical production domain, e.g. `https://example.com`)
- `preferred_host` (`example.com` vs `www.example.com`)
- `deployment_envs` (prod/stage/local URLs if applicable)

**SEO Essentials**
- `default_title_template` (e.g. `{Page} | {Brand}`)
- `default_meta_description` (brand-level)
- `primary_language` (e.g. `en`)
- `target_countries` (optional)
- `indexing_policy` (index/noindex rules for environments)

**Social/Sharing**
- `default_og_image` URL (absolute)
- `twitter_handle` (optional)
- `brand_logo` file path or URL (optional, for visual checks)

**Analytics**
- Analytics vendor(s) + IDs (optional)
- Consent requirements (GDPR/CCPA) (if applicable)

> Tool behavior: If values are missing/ambiguous, the tool must ask yes/no confirmation and accept corrections.

## 2) Definitions: severities + required evidence

Every issue must include:
- **Severity**: `Blocker` / `High` / `Medium` / `Low`
- **Evidence**: screenshot (if visual), HTML snippet, URL(s), lighthouse/axe output, or file path + line(s)
- **Fix guidance**: exact change recommendation; if code mode is enabled, include file targets.

Severity meanings:
- **Blocker**: prevents usage, breaks layout on common devices, or critical SEO/indexing/security issue
- **High**: materially harms conversion, accessibility, SEO, or performance
- **Medium**: best practice violations that add risk or degrade quality
- **Low**: minor improvements / polish

## 3) Visual + responsive (core WebSeed UX)

### 3.1 Required viewport testing (visual proof)
The tool must capture screenshots of key pages at minimum:
- **Mobile**: 390×844 (or 375×812)
- **Tablet**: 768×1024
- **Desktop**: 1440×900

For each viewport, check:
- No horizontal scrolling (unless intentional for carousels)
- No clipped text/buttons
- Navigation usable (menu accessible, tap targets OK)
- Consistent spacing and typography scale

### 3.2 Mobile spacing/padding standards (practical rules)
Mobile responsiveness is often ruined by excessive padding. Use these constraints:

- **Default page gutters** (content padding left/right):
  - Mobile: **16px** (acceptable range 12–20)
  - Tablet: **24px**
  - Desktop: **32–64px** depending on layout
- **Avoid “double padding”**: don’t pad both outer container and inner sections heavily.
- **Avoid giant vertical padding** on mobile:
  - Typical section vertical padding: **24–48px**
  - Hero vertical padding: **32–72px** (rarely more on mobile)
- **Line length**:
  - Body text should target ~45–90 characters per line on desktop; avoid ultra-wide text blocks.
- **Touch targets**:
  - Minimum **44×44px** tap area for buttons/controls.

> Audit heuristics (visual + DOM): flag sections where computed padding causes content width < ~280px on typical mobile, or where above-the-fold contains too little information due to padding.

### 3.3 Typography and layout consistency
- Use a consistent type scale (e.g. 16px base on mobile)
- Avoid tiny text (<14px) for core content
- Ensure adequate contrast (see accessibility)
- Avoid layout shift caused by late-loading fonts/images

### 3.4 Media rules
- Images must be responsive (`srcset`/`sizes` if possible)
- Avoid serving huge images to mobile
- Provide explicit `width` and `height` (or CSS aspect-ratio) to reduce CLS
- Prefer modern formats (AVIF/WebP) where possible
- Lazy-load offscreen images (but never lazy-load the LCP hero image)

## 4) Accessibility (non-negotiable)

### 4.1 Minimum compliance target
- Aim for **WCAG 2.2 AA** for public-facing sites.

### 4.2 Must-pass checks (Blocker/High)
- Page has exactly one `h1` (typical) and logical heading order (no random jumps)
- All images have meaningful `alt` (decorative images must have empty alt)
- All interactive controls are reachable via keyboard
- Visible focus states (not removed)
- Color contrast meets AA for text and UI controls
- Form fields have labels; errors are announced and understandable
- No “keyboard traps”
- `lang` attribute set correctly

### 4.3 Tooling expectations
- Run an automated scan (e.g., axe) on key pages
- Report: rule ID, selector, impact, and screenshot if relevant

## 5) Technical SEO (crawl/index/canonical correctness)

### 5.1 Indexing + environment safety
- Production pages: indexable by default unless explicitly excluded
- Staging/local must be `noindex` (or blocked) to prevent accidental indexing
- `robots.txt` present and correct
- XML sitemap present and references canonical URLs

### 5.2 Canonical + host consistency (Blocker/High)
- Exactly one canonical URL per page (absolute URL)
- Canonicals must match the confirmed `primary_domain` and preferred host
- HTTP → HTTPS redirect
- Non-preferred host → preferred host redirect (www vs non-www)
- Trailing slash policy consistent (pick one)

### 5.3 Metadata requirements per indexable page
**Required**
- `<title>`: unique, meaningful, not truncated (roughly 30–60 chars guideline)
- Meta description: present (often), not duplicated site-wide; avoid spam
- `meta viewport` present
- OpenGraph: `og:title`, `og:description`, `og:image`, `og:url`
- Twitter card tags if needed

**Structured Data**
- Use JSON-LD where applicable:
  - Organization/LocalBusiness (if relevant)
  - WebSite + SearchAction (if relevant)
  - Article/BlogPosting for content pages
  - Product/FAQ (only if content matches guidelines)

### 5.4 Content SEO basics (quality)
- Clear H1 describing the page intent
- Headings structure supports skimming
- Internal links are descriptive (avoid “click here”)
- Avoid thin/duplicate pages
- 404 page exists and is helpful; returns real 404 status

## 6) Performance (Core Web Vitals + practical budget)

### 6.1 Performance targets (guidelines)
Targets vary, but the WebSeed standard should aim for:
- **LCP**: ≤ 2.5s (good), ≤ 4.0s (needs improvement)
- **CLS**: ≤ 0.1
- **INP**: ≤ 200ms (good), ≤ 500ms (needs improvement)
- Keep JS/CSS lean; avoid shipping large unused bundles

### 6.2 Must-check items
- Compression enabled (brotli/gzip)
- Caching headers sane for static assets
- Avoid render-blocking resources (or justify)
- Critical images optimized
- Fonts:
  - `font-display: swap`
  - preload only if beneficial, don’t over-preload
- Third-party scripts minimized; tag manager bloat flagged

### 6.3 Tooling expectations
- Run Lighthouse (or equivalent) and record:
  - performance/seo/accessibility/best-practices categories
- Provide actionable items, not just scores

---

## 7) Security + privacy baseline

- HTTPS only; no mixed content
- Security headers (recommended):
  - `Content-Security-Policy` (at least a reasonable baseline)
  - `X-Content-Type-Options: nosniff`
  - `Referrer-Policy`
  - `Permissions-Policy` (as appropriate)
- Cookies:
  - Set `Secure`, `HttpOnly` (where applicable), and proper `SameSite`
- If collecting personal data:
  - Privacy policy accessible
  - Consent controls where legally required

---

## 8) Content and UI quality checks

### 8.1 Broken links and assets
- No broken internal links (4xx/5xx)
- External links should not be systematically broken
- Images/scripts/css should not 404
- Add link checks to the audit

### 8.2 UI consistency checks
- Buttons have consistent styles
- Hover/active/disabled states defined
- Navigation consistent across pages
- Forms provide success + error states

### 8.3 Internationalization (if applicable)
- `hreflang` correctly configured
- Language switcher is crawlable and consistent
- Avoid duplicate content across locales without correct hreflang/canonicals

---

## 9) Code quality standards (only if source code is provided)

### 9.1 No-surprise modifications policy
- The tool **must not modify code** unless the user explicitly approves.
- Default mode: report-only.
- Autofix mode: generates a patch/diff + asks for confirmation before applying.

### 9.2 Repository hygiene
- Clear build/run instructions
- Environment variables not committed
- Lint/format configured (project dependent)
- Dead code/unused dependencies flagged (Medium)

### 9.3 Frontend implementation basics
- Semantic HTML (`header`, `nav`, `main`, `footer`)
- Avoid div-only pages
- Images use correct `alt`
- Avoid inline styles unless justified
- Avoid fixed heights that break mobile layouts

---

## 10) Audit scope: what pages must be checked

Minimum:
- Home
- One content-heavy page (about/blog/article)
- One conversion page (pricing/contact/signup)
- 404 page (if possible)
- Any key landing pages

For each, collect:
- URL
- status code
- canonical
- title/description
- H1
- screenshot(s)
- lighthouse/axe results
- detected issues
